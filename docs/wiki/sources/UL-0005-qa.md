# Q&A — UL-0005 초안 보완 (레거시 잔재 정리 근거 보완)

- 관련 UL: UL-0005 (새 프로젝트(job-fresh) 착수 전 레거시 잔재 정리)
- 질문 배경: lint-ul 파이프라인 내부 검사에서, UL-0005 초안의 다음 3개 결정에 근거(Basis)가 비어 있다는 점이 지적됨: (1) 처리 한도 문장 삭제 시 원래 근거였던 "API 과금 대응" 원칙의 향방, (2) zighang_weekend 기능 폐기의 실질 사유, (3) 원티드 배제에 따른 Jina Reader 폐기가 직행 API 자체의 본문 추출 실패 시 폴백 처리에 미치는 영향.

**Q1. living-spec.md "실행당 처리 한도: 직행 8건, 원티드 2건 (API 과금 대응)" 문장을 삭제하는데, 원래 근거였던 "비용 통제(API 과금 대응)"는 이제 어떻게 되는가?**

A1. 새 코드에서 재설계 예정이며 아직 미결이다. 비용 통제 필요성 자체를 부정하는 것이 아니라, 구체적인 수치(직행 8건 등)를 지금 living-spec에 다시 못박지 않고 재작성 시점으로 미룬다.

**Q2. keywords.json의 zighang_weekend(요일별 지역 필터) 기능을 폐기하는 실질 사유는 무엇인가?**

A2. 요일별로 지역 필터를 다르게 가져갈 필요 자체가 없다고 판단했다(요일별 지역 분기 자체가 불필요).

**Q3. 원티드를 배제하면서 Jina Reader도 폐기하는데, 직행 API 자체의 본문 추출이 실패했을 때의 폴백은 어떻게 하는가?**

A3. 폴백 없이 실패로 처리한다 — 해당 공고는 건너뛴다.
