# 회의록 — 온보딩 장벽 해소를 위한 배포 아키텍처 결정

- 일시: 2026-07-08
- 참석자: 이주엽(프로젝트 제작자) / 자문 — Backend Architect, DevOps Automator, Software Architect (엔지니어 에이전트 3인)

## 초기 고민

제작자 본인은 이미 세팅이 끝난 상태로 Gongtion을 편하게 쓰고 있으나, 주변 소프트웨어학과 친구들에게 사용해보라고 권하기엔 초기 세팅 절차가 너무 많아 이탈률이 매우 높을 것으로 예상됨. 현재 절차: GitHub 저장소 Fork → Notion DB를 정확한 스키마로 생성(Property 이름·타입 하나라도 틀리면 안 됨) → Notion Integration 생성 및 API 키 발급 → DB에 Integration 연결 → keywords.json에 본인 조건 직접 편집 → Gemini API 키 발급 → 위 모든 키를 GitHub Secrets에 개별 등록 → Fork 저장소는 Actions가 기본 비활성화라 수동으로 켜야 함. "자동화(등록만 하면 알아서 주기적으로 돈다)"라는 컨셉은 유지하면서 이 진입장벽만 낮출 방법을 결정하고자 함.

## 최종 결정 사항

1. **실행 모델**: 분산형 fork 구조 유지(중앙 실행 모델로 전환하지 않음). 이유: 중앙화는 친구들의 API 비용·장애 대응을 제작자 혼자 떠안는 비가역적 결정이며, 지인 대상·상용 SLA 없는 개인 프로젝트 규모에는 과도한 운영 부담.
2. **저장소**: Notion DB → 각 fork 저장소 내부 Git 파일 저장으로 전환. Notion 관련 셋업 절차(DB 생성·Integration·API 키 발급·연결) 전체 제거. Notion의 "보기 좋은 DB 한 줄 정리" UX를 잃는 단점은 미약하다고 판단.
3. **AI 기능**: 파이프라인의 필수 코어에서 분리해 옵션 기능으로 재배치. 운영자가 소유한 AWS Lambda 공용 프록시가 Gemini 호출을 대행하여, 사용자 쪽에는 AI 관련 시크릿이 필요 없어짐(운영자의 Gemini 키는 Lambda 환경변수에만 존재).
4. **레이트리밋 대응**: 5명 내외 사용자의 크론이 비슷한 시간대(화·수·토 07:00 KST 근처)에 몰려 Gemini 무료 티어 레이트리밋(분당 15회)에 걸리는 문제를 다음 조합으로 해소 — 크론 지터(온보딩 스크립트가 ±0~25분 랜덤 오프셋 자동 배정) + Lambda 호출 직전 0~5분 랜덤 sleep + Lambda Reserved Concurrency 1~2로 전역 직렬화 + Lambda 내부 토큰버킷(분당 12회 자체 상한)과 429 시 지수 백오프 재시도(1s→2s→4s, 2~3회) + 여러 공고를 배치로 묶어 호출 수 절감 + 재시도까지 실패하면 요약 없이 진행(최후 폴백). 이 규모(5명, 약 1개월 운영)에서 SQS 등 큐 인프라는 과설계로 판단해 도입하지 않음.
5. **인증**: 사용자 fork에는 AI 관련 시크릿을 요구하지 않는 것이 목표. GitHub OIDC(Actions `id-token: write` + API Gateway JWT Authorizer로 issuer/repository claim 검증) 방식을 우선 검토하고, 구현 난이도가 높으면 저위험 정적 프록시 키(Gemini 키와 무관) 하나만 Secrets에 등록하는 절충안을 대안으로 둠.

이 결정은 기존에 README.md 기준으로 기록된 베이스라인(Notion 저장, 사용자별 Notion/Gemini API 키 개별 발급 및 등록)에서, "온보딩 이탈률이 너무 높다"는 문제의식에 따라 저장소와 AI 처리 방식을 변경하는 것이며, 자동화라는 핵심 컨셉 자체는 변경하지 않는다.
