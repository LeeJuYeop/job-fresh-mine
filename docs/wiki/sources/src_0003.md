# 회의록 — keywords.json 입력 장벽 해소 및 온보딩 스크립트 최종 설계

- 일시: 2026-07-08
- 참석자: 이주엽(프로젝트 제작자) / 자문 — Backend Architect, DevOps Automator, Software Architect (엔지니어 에이전트 3인)

## 초기 고민

온보딩 장벽 문제(저장소 Notion→Git 전환, AI Lambda 프록시 도입)는 이미 별도 결정으로 확정된 상태에서, `keywords.json` 설정 자체의 입력 난이도가 남은 마찰 지점으로 지적됨. `zighang`/`zighang_weekend` 블록의 `depthTwos`(직무), `regions`(지역), `employeeTypes`(채용유형) 값은 crawler.py에서 검증 없이 그대로 직행(zighang) 검색 API의 쿼리 파라미터로 전달되므로, 사용자가 입력하는 값이 직행 사이트 내부의 원본 태그 문자열과 정확히 일치해야 함(예: `"DevOps_SRE"`, `"서버_백엔드"` — 대소문자·언더바까지 정확히 타이핑 필요, 오탈자 시 조용히 빈 결과). 사용자가 "백엔드, DevOps처럼 원하는 것을 자연스럽게 표현하면 정확한 태그로 자동 변환되는 방식"을 원함.

## 최종 결정 사항

1. **온보딩 스크립트 도입**: `scripts/setup.py`라는 로컬 Python 스크립트를 신설한다. 사용자는 Fork → Clone 후 이 스크립트를 1회 실행하는 것만으로 설정을 마친다.
2. **태그 수집 방식**: 스크립트 실행 시점에 직행 API로 필터 없는 조회를 보내 현재 유효한 `depthTwos`/`regions`/`employeeTypes` 태그 전체를 실시간으로 수집한다(코드 확인 결과 페이지네이션 없는 단일 요청 수준이라 직행 서버에 부담 없음). 태그 매핑을 코드에 영구 하드코딩하지 않는 이유는, 직행이 태그 체계를 바꾸면 하드코딩이 조용히 노후화되어 지금과 같은 실패가 "사용자 오탈자"에서 "코드 노후화"로 자리만 옮기기 때문. 크론(정기 실행)에는 이 조회 로직을 넣지 않고 setup 시점에만 수행해, 정기 자동화에 새 런타임 의존성을 추가하지 않는다.
3. **체크박스 picklist**: 수집한 태그를 사람이 읽을 수 있는 형태로 보여주고 체크박스로 선택받아, 선택 결과를 그대로 원본 태그 문자열로 `keywords.json`에 기입한다. 사용자는 원본 태그 문자열을 직접 타이핑하지 않는다.
4. **저장 방식은 plain git**: 생성된 `keywords.json`을 `git add/commit/push`로 자동 반영한다. 사용자가 이미 clone 시점에 보유한 git 자격증명(credential helper/SSH)을 그대로 재사용하며, 별도 토큰을 스크립트에 내장하지 않는다. push 전 `origin`이 업스트림 원본이 아니라 본인 fork를 가리키는지 확인하는 안전장치를 둔다.
5. **`gh` CLI 의존성 완전 배제**: 애초 검토했던 `gh api`(Actions 자동 활성화), `gh secret set`(시크릿 자동 등록), `gh workflow run`(셀프테스트 자동화)을 모두 스크립트에서 제외한다. 이유: 스크립트가 사용자 GitHub 계정에 대한 쓰기 권한(Secrets, Actions 상태 변경)을 갖는 순간, 스크립트 결함 하나가 여러 사용자 계정에 반복적으로 영향을 미치는 위험한 자동화가 되기 때문. 이를 사람이 웹 UI에서 직접 누르는 수동 조작으로 남기는 편이 약간의 마찰과 위험도를 저울질했을 때 더 타당하다고 판단.
6. **Actions 활성화는 수동**: GitHub 웹 UI의 Actions 탭에서 2클릭으로 처리(기존 README에도 이미 있던 절차라 추가 부담 없음).
7. **AI 관련 시크릿은 GitHub OIDC로 대체**: OIDC 구현을 확정하여, 애초에 시크릿을 등록할 대상 자체가 없어진다(별도 자동화 불필요).
8. **셀프테스트는 자동화하지 않음**: 필요 시 사용자가 Actions 탭에서 수동으로 "Run workflow"를 클릭해 확인할 수 있는 옵션으로만 남긴다. 다만 "Actions 비활성화" 상태는 에러 로그조차 남지 않는 침묵 실패이므로, 이를 메우기 위해 (a) README에 트러블슈팅 섹션("설정 후 며칠 지나도 새 커밋이 없다면 Actions 탭에서 활성화 여부를 확인하라")을 추가하고, (b) `setup.py` 실행 종료 시 "다음 할 일 체크리스트"(Actions 활성화 링크, 수동 실행 권장)를 텍스트로 출력한다.

이 결정으로 사용자가 실제로 수행하는 절차는 Fork → Clone → `scripts/setup.py` 실행(체크박스 선택) → (수동) Actions 탭에서 2클릭 활성화, 총 네 동작으로 축소된다.

## 추가 논의 — profile.json 입력 마찰

### 초기 고민

`profile.json`(선택 기능인 개인화 분석용 프로필)도 온보딩 마찰 지점이지만, keywords.json과 성격이 다름. `tech_stack`(보유 기술), `learning_interests`, `projects`(이름·설명·사용기술)는 직행처럼 고정된 태그 목록이 존재하지 않는 **자유 텍스트/개인 데이터**라, keywords.json에 적용한 체크박스 picklist 방식이 통하지 않음. 사용자가 자기 정보를 어떤 형태로든 입력하는 최소 비용 자체는 없앨 수 없는 문제.

### 최종 결정 사항

1. **같은 `scripts/setup.py`에 통합**: 별도 도구를 새로 만들지 않고, keywords.json 단계 다음에 이어지는 opt-in 섹션으로 추가한다.
2. **opt-in 질문**: "개인화 매칭 기능을 쓰시겠습니까? (y/N)"을 먼저 물어, 원치 않으면 이 섹션 전체를 건너뛴다. 이미 정의된 "파일이 없으면 개인화 기능 자동 비활성화" 동작과 자연스럽게 맞물림.
3. **리스트형 입력(tech_stack, learning_interests)**: 쉼표로 구분한 자유 텍스트 한 줄 입력을 받아 `split(",")` 후 trim만 하고 그대로 저장. 정답이 없는 개인 데이터이므로 keywords.json과 달리 화이트리스트 대조 검증은 하지 않는다.
4. **projects(반복 입력)**: 이름/설명/사용기술 3개를 묻는 질문을 `while` 루프로 반복하고, 매 라운드 후 "더 추가하시겠습니까?"로 종료 조건을 두거나 빈 입력 시 자동 종료한다.
5. **JSON 문법 오류 원천 차단**: 사용자가 파일을 직접 편집하게 하지 않고 스크립트가 대화형 입력을 받아 `json.dump`로 생성. 생성 직후 `json.load`로 재검증하는 자기 검증 단계를 추가한다.
6. **재편집 소유권은 사용자에게**: profile.json은 Notion 스키마 같은 외부 API 연동 제약이 없는 순수 로컬 파일이므로, 스크립트는 최초 1회 스캐폴딩만 돕고 이후 재편집은 사용자가 직접 텍스트 에디터로 하도록 안내한다(재실행 시에는 기존 파일이 있으면 "덮어쓸까요/건너뛸까요"만 확인). 이를 보완하기 위해 주석이 달린 `profile.json.example` 샘플 파일을 별도로 두어, 스크립트를 건너뛰고 나중에 직접 편집하고 싶은 사람도 스키마를 추측할 필요가 없게 한다.
7. **프라이버시 경고**: fork 저장소가 public이면 프로젝트 설명 등 개인 데이터가 git 히스토리에 그대로 노출되므로, README에 이에 대한 경고 문구를 추가한다.

이 결정은 "keywords.json은 정답이 있는 매칭 문제, profile.json은 정답이 없는 자기 서술 문제"라는 성격 차이를 인정하면서도, 이미 존재하는 setup.py에 몇 개의 opt-in 질문을 추가하는 것은 한계 비용이 낮다고 판단하여 두 접근(대화형 스크립트 통합 vs 문서화만으로 충분)을 절충한 것이다.
