# 회의록 — AI 요약 2차 가공에 Claude 루틴 도입 검토

- 일시: 2026-07-09
- 참석자: 이주엽(프로젝트 제작자) / 자문 — Backend Architect, DevOps Automator, Software Architect (엔지니어 에이전트 3인, 1차 독립 검토 후 상호 반박까지 진행)

## 초기 고민

AI 요약(구조화·한줄요약·개인화 매칭)을 현재 계획된 AWS Lambda 공용 프록시 대신, **각 사용자가 개인 환경에 설정하는 Claude 루틴(Claude Code의 자체 스케줄 실행 기능)**으로 옮길 수 있는지 검토함. 구상: 크롤링(직행/원티드 API 호출)은 Claude 토큰을 과도하게 소모하므로 기존대로 GitHub Actions가 맡아 1차 소스를 Git에 저장하고, 그 소스를 각 사용자의 Claude 루틴이 읽어 자신의 profile.json 기반으로 2차 AI 가공을 수행. 가공 결과 저장 위치는 Git 외에 MCP 연결로 Notion·Slack도 자유롭게 선택 가능. 운영자의 루틴이 여러 사용자 Git에 접근하는 중앙집중 방식은 배제하고, 도입한다면 Claude 루틴은 각 사용자에게 선택적 가이드라인으로만 제공하는 구조를 전제로 함.

이 도입이 프로젝트에 미치는 두 가지 쟁점을 검토 대상으로 삼음:
1. 기술적으로 실현 가능하며 의도대로 동작하는가(데이터 정합성, 상태 추적, 실패 처리, 기존 Lambda 방식과의 관계).
2. 도입 시 프로젝트 정체성이 "서비스 제공"에서 "Git+Claude 루틴 활용법 공유"로 이동하는데, 이것이 제작자가 이 프로젝트 운영에서 얻고자 하는 목적(실사용자 피드백이 있는 프로젝트 운영 경험)과 맞는가.

## 최종 결정 사항

1. **현재는 도입하지 않는다.** Claude 루틴을 AI 가공 경로로 코드·인프라에 반영하는 것은 보류한다. 이유는 두 갈래:
   - **정체성 관점**: 분산 fork 구조에서 운영자가 관찰·개입할 수 있는 지점은 현재 AWS Lambda 프록시가 유일하다(이 지점이 프로젝트를 "레시피 공유"가 아닌 "서비스 운영"으로 만드는 근거). Claude 루틴은 이 마지막 지점을 없애 운영자를 관찰 불가능한 상태로 만든다.
   - **운영 관점**: 도입이 해결하는 문제(Gemini 레이트리밋 몰림 대응)가 애초에 5명·1개월 규모에서는 발생 가능성이 낮은 문제로 판단됨(크론 지터 정도로 충분히 회피 가능). 반대로 도입 비용(온보딩 마찰 2단계 추가, 사용자별 환경 편차, 중앙 관측성 상실)은 지금 즉시 실재한다. 확실한 손실로 불확실한 이득을 사는 거래로 판단.
2. **관측성 붕괴는 opt-in 텔레메트리로 메워지지 않는다.** 각 사용자 루틴이 상태를 리포팅하게 하는 방안을 검토했으나, "신호 없음"이 실패인지 정상 유휴인지 구분할 수 없는 근본 한계가 있어 반쪽짜리 대책으로 결론. 제대로 하려면 dead-man's switch급 중앙 감시가 필요한데, 이는 없애려던 중앙 인프라를 다시 짓는 것과 같아 실익이 없음.
3. **두 AI 경로를 동시에 코드로 병행(어댑터 공존)하지 않는다.** 아키텍처적으로는 크롤링(Git)과 AI 가공을 분리해 컨슈머를 교체 가능하게 만드는 설계 자체는 타당하나, 실사용자가 사실상 한 경로(Lambda)뿐인 지금 규모에서 두 번째 경로는 검증 트래픽도 없이 유지 비용만 지는 죽은 추상화로 판단. 택일(Lambda 단일 경로 유지)이 맞다.
4. **README에 "레시피" 형태로 언급하는 것은 허용한다.** "Claude 구독자는 프록시 대신 이 Git 소스에 직접 루틴을 붙여 개인 가공을 할 수도 있다"는 안내 문단은 코드 결합이 없어 완전히 가역적이고, 정체성상의 "활용법 공유" 측면을 낮은 비용으로 담아낼 수 있다. 단, 이 문단에는 **1차 소스 Git 파일 스키마가 "호환성 비보장(best-effort)"임을 반드시 명시**한다 — 그래야 향후 스키마를 자유롭게 바꿀 수 있고, 명시하지 않으면 의도치 않게 사실상의 API 호환 의무를 지게 된다.
5. **Claude 루틴 재검토 조건을 명시해둔다.** 향후 사용자 규모가 커져 운영자가 부담하는 Gemini 비용·거버넌스가 실제 문제가 될 때 이 논의를 다시 꺼낸다. 이때도 도입 명분은 "레이트리밋 회피"가 아니라 "AI 비용·소유권을 사용자에게 분산"이어야 한다.
6. **AI 가공과 무관하게 지금 바로 반영할 가치가 있는 것 2가지**를 별도로 확인함(Claude 루틴 도입 여부와 무관한 순이득):
   - 1차 소스 Git 파일의 **스키마 버전 고정**: 저장소가 Notion에서 Git 파일로 전환된 시점부터 이 파일 포맷은 분산된 각 fork에 대한 사실상의 API가 되었으므로, 버전 표기 없이 필드를 바꾸면 사용자 fork가 조용히 깨질 수 있음.
   - 공고 레코드에 **안정적 고유 ID 부여**(예: 공고 URL 해시): dedup·재처리·향후 어떤 형태의 2단계 파이프라인 분리에도 전제가 되는 저비용 조치.
7. **레이트리밋 대응 인프라(UL-0002에서 계획한 토큰버킷·Reserved Concurrency 등)를 지터+지수 백오프 중심의 최소 구성으로 다이어트하는 것을 별도 검토 과제로 남긴다.** 5명 규모에서 기존 계획 자체가 다소 과설계였을 수 있다는 지적이 있었으나, 이번 논의의 확정 결정 사항은 아니며 추후 별도로 다룬다.

이번 결정으로 AI 요약 파이프라인은 UL-0002/UL-0003에서 확정된 AWS Lambda 프록시 구조를 그대로 유지하며, Claude 루틴은 코드가 아닌 문서(README 레시피)로만 선택지를 열어둔다.
