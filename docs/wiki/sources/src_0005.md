# 작업 지시 — 새 프로젝트 착수 전 레거시 잔재 정리

- 일시: 2026-07-09
- 작성자: 이주엽(프로젝트 제작자) — 회의가 아닌 채팅을 통한 직접 작업 지시

## 배경

새 프로젝트(job-fresh)를 본격적으로 시작하기 전에, 기존 프로젝트(Gongtion) 시절의 이름·용어·플랫폼 범위·문서 서술이 남아 있으면 새 설계와 계속 충돌하므로, 이를 먼저 걷어내는 데 집중하기로 함. 아래 4가지를 결정 사항으로 지시함.

## 결정 사항

### 1. 프로젝트명 변경
`Gongtion` → `job-fresh`로 프로젝트명을 변경한다.

### 2. 용어 변경: 크롤링 → API
현재 프로젝트는 사실상 크롤링(스크래핑)보다 REST API 호출에 가깝다. 이에 따라 "크롤링"이라는 용어를 "API"로 바꾸고, 관련 파일명도 다음과 같이 변경한다.
- `crawler.py` → `zighangApi.py` — **완료**(이미 파일 리네임 반영됨. 원래 crawler.py 내용은 `legacy_crawler.py`로 별도 보존).
- `.github/workflows/crawler.yml` → `job-fresh-pipeline.yml` — **결정 사항(예정)**. 아직 디스크에는 반영되지 않았고, 현재는 원본이 `legacy_crawler.yml`로만 보존되어 있는 상태. 실제 파일 생성/커밋은 이후 별도 작업으로 진행한다.

### 3. 플랫폼 변경: 원티드 배제, 직행 단일화
현재 직행(공식 API)과 원티드(비공식 JSON API) 두 플랫폼을 사용 중이나, 원티드는 일단 배제하고 직행에만 집중한다.
- **근거**: 직행 하나에 집중해 새 프로젝트의 개발 범위를 축소하기 위함(원티드 API 자체의 불안정성이 주된 사유는 아님).
- **파급 효과 1 — living-spec.md에서 원티드 관련 서술 전체 삭제**: 기능 섹션의 "원티드(비공식 JSON API)에서 공고 수집" 언급, AI 구조화 설명 중 원티드 전용 추출 필드(직무·경력·채용유형) 언급, 아키텍쳐 섹션의 원티드 처리 흐름(공고 목록 수집 → 상세 API 본문 추출 → AI 프록시로 직무·경력·채용유형·기술스택 추출 → Git 저장) bullet 전체, 기술스택 섹션의 "원티드 비공식 JSON API" 항목이 모두 삭제 대상이다.
- **파급 효과 2 — Jina Reader 폐기**: 원티드를 사용하지 않게 되면서 Jina Reader(본문 추출 폴백)도 더 이상 사용하지 않는다. living-spec.md 아키텍쳐 섹션의 "API 본문 추출 실패 시 Jina Reader로 자동 폴백" bullet과, 기술스택 섹션의 "Jina Reader (본문 추출 폴백)" 항목이 삭제 대상이다.

### 4. living-spec.md 세부 서술 삭제 (아키텍쳐 섹션 line 29, 30)
아래 두 문장은 living-spec.md에서 삭제한다.
- (line 29) "실행당 처리 한도: 직행 8건, 원티드 2건 (API 과금 대응)." — 원티드 배제로 원티드 관련 수치는 물론, 처리 한도 서술 자체를 삭제한다.
- (line 30) "`keywords.json`의 `zighang`/`zighang_weekend` 필터를 상황별로 다르게 적용 가능하며, 스케줄·필터 전환 조건은 GitHub Actions 설정으로 커스터마이징 가능." — 이 문장을 삭제하는 것은 단순 서술 정리가 아니라, **`zighang_weekend`(요일별 지역 필터) 기능 자체를 폐기**하는 것을 의미한다. `keywords.json`의 `zighang_weekend` 섹션과 `legacy_crawler.py`의 요일별 분기 로직(`resolve_zighang_cfg`)도 함께 폐기 대상이며, 이후 `zighang` 단일 설정만 사용한다.
