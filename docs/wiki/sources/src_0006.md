- 참석자: 이주엽(프로젝트 제작자) / 자문 — Code Reviewer (엔지니어 에이전트, 레거시 코드 이행 리뷰 수행)

## 초기 고민

UL-0005(레거시 잔재 정리) 확정 후 실제 코드 이행에 착수하기 전, Code Reviewer 에이전트에게 레거시 코드(`legacy_crawler.py`, `legacy_pipeline.py`, `legacy_crawler.yml`)를 위키 계획과 대조 분석시킨 결과, 레거시에 대응물이 없어 **구현 전에 결정이 필요한 설계 공백 8개**가 식별됨(Git 저장 포맷, 고유 ID, dedup·재시도, Lambda 프록시 계약, 처리 한도, 커밋/푸시 방식, OIDC 검증, AI 실패 표식, keywords.json 스키마). 이를 하나씩 결정하고, 논의 중 AI 가공 범위 축소(기술스택 추출 폐지) 결정이 추가로 내려짐.

## 최종 결정 사항

### 1. 탐색 주기 및 조건 (수집 방식 변경)

- **매일 실행으로 변경**: 기존 화·수·토 07:00 KST 크론을 폐기하고 매일 실행한다.
- **당일 등록 공고만 조회**: 직행 API가 당일 올라온 공고 조회를 지원함을 확인했고, 매 실행은 당일 등록분만 가져온다.
- 사용자가 keywords.json으로 고른 직무·경력 등 필터링 원칙은 그대로 유지.
- **탐색 개수 제한: 하루 20건(잠정)**. 확정 수치는 추후 별도로 결정한다(UL-0005의 처리 한도 Pending은 이 잠정치로 운영하되 계속 열어둠).

### 2. 중복 확인(dedup) — 사실상 폐기

- "당일 등록분만 조회" 조건 덕분에 실행 간 중복이 구조적으로 발생하지 않으므로, **별도 dedup 로직을 두지 않는다**.
- 동일 공고를 여러 날에 걸쳐 재게시하는 경우는 거르지 못하지만, 이 한계는 인지한 채 수용하고 넘어간다.

### 3. Git 저장 포맷

- **공고당 1파일**, YAML frontmatter + 마크다운 본문 하이브리드.
- **저장 경로: `jobs/{YYYY-MM-DD}/` 일별 폴더**. 일별로 구분하지 않으면 사용자가 며칠간 확인을 미뤘을 때 확인한 공고와 안 한 공고를 구분할 수 없음 — 일별 폴더가 이를 해결.
- **파일명: `[회사명][경력][채용유형][지역] {title}.md`** — 파일명에서 최대한 많은 정보를 표시. 부속 규칙: Windows 금지문자(`\/:*?"<>|`) 제거 및 길이 제한 처리, 복수 지역 공고는 첫 지역만 파일명에 표기(전체 지역은 frontmatter에 기록).
- frontmatter에 `schema_version`을 명시(UL-0004 확정 사항 이행).

### 4. 삭제 원칙 (보존 기간)

- **14일 이상 지난 파일은 삭제를 원칙**으로 한다. 폴더명에 날짜가 명시되므로 기간이 지난 일별 폴더를 폴더째 삭제한다. (삭제는 매 실행 시 워크플로가 수행)

### 5. 고유 ID (UL-0004 이행 방식)

- frontmatter의 `id` 필드에 **`zighang-{직행 recruitment UUID}`** 형식으로 기록. 직행 API가 이미 제공하는 UUID에 소스 접두사만 부착 — 해시 계산 불필요, URL과 사람이 대조 가능, 향후 플랫폼 확장 여지 확보.
- dedup을 쓰지 않게 되었으므로 ID의 용도는 재처리·추적·Claude 루틴 레시피의 전제로 축소됨.

### 6. AI 가공 범위 축소 — 기술스택 추출 폐지

- **AI는 한줄요약(ai_comment)만 수행한다. 기술스택 추출은 하지 않는다.**
- 사유: 기술스택 추출은 원래 Notion DB의 multi-select 속성으로 예쁘게 보이게 하려던 것인데, Notion 저장이 폐기된 지금은 필요 없음. 본문에 이미 있는 내용을 굳이 한 번 더 정리할 이유가 없음.
- 개인화 매칭(personal_comment, profile.json 기반)은 기존 결정대로 유지.

### 7. Lambda AI 프록시 계약

- **프롬프트는 Lambda가 소유**한다. fork는 공고 데이터(+ 선택 시 profile)만 구조화 JSON으로 전송 — 임의 프롬프트로 Gemini를 무단 사용하는 남용을 차단하고, 프롬프트 버전을 운영자가 일괄 관리(UL-0004의 "운영 시임" 논리와 정합).
- **profile.json 전송 허용**: 개인화 매칭을 위해 profile 데이터가 운영자 인프라(Lambda)를 경유하는 것을 허용하되, Lambda가 이를 로깅하지 않는다는 무로깅 원칙을 README에 명시한다.

### 8. GitHub Actions 커밋/푸시 방식

- 기본 제공 **`GITHUB_TOKEN` + `permissions: contents: write`**로 기본 브랜치에 직접 커밋·푸시. 추가 시크릿 불필요("사용자 fork 측 Secrets 없음" 원칙 유지).
- 동시 실행 충돌 방지: workflow `concurrency` 그룹 설정 + push 전 `git pull --rebase`.
- 일별 폴더 생성과 14일 경과 폴더 삭제도 이 커밋에 포함된다.

### 9. OIDC 검증 규칙

- **운영자가 관리하는 허용 리스트** 방식. 사용자 계정 목록을 Lambda 환경변수 등으로 관리 — 이름만 맞춘 무관 저장소의 무임승차(운영자 Gemini 비용 잠식)를 차단. 5명 내외 규모에서 한 줄 추가 비용은 무시 가능. 규모 확장 시 패턴 매칭으로 완화 검토.

### 10. AI 실패 표식

- frontmatter에 **`ai_status: done | skipped`** 필드를 둔다. AI 프록시 최후 폴백("요약 없이 진행", UL-0002)으로 저장된 레코드를 식별 가능하게 함.
- 재처리(backfill) 파이프라인은 구현하지 않는다(현 규모에 과설계). 향후 필요 시 이 필드 기반으로 확장 가능. 14일 삭제 원칙상 재처리 기회도 14일로 자연 제한됨.

### 11. keywords.json 새 스키마

- **평탄화 + `schema_version`**: 원티드용 `keywords` 배열과 `zighang_weekend` 블록(UL-0005에서 폐기 확정)이 사라진 뒤, `zighang` 래핑도 제거하고 최상위에 필터 필드를 직접 둔다. 직행 단일화가 확정이므로 래핑이 불필요.

```json
{
  "schema_version": 1,
  "depthTwos": ["서버_백엔드", "DevOps_SRE"],
  "regions": [],
  "employeeTypes": [],
  "careerMin": 0,
  "careerMax": 0,
  "educations": []
}
```
