## 배경

UL-0006 확정 이후 `zighangApi.py`·`pipeline.py`를 실제로 구현하고 GitHub Actions에서
실행·검증하는 과정에서, 설계 단계에서는 드러나지 않았던 4가지 사항이 정정·추가됨.

## 최종 결정 사항

### 1. 오케스트레이션 주체 반전 + `src/` 폴더 구조

- **기존 구현(1차 작성)**: `zighangApi.py`가 진입점(`main()`)으로서 설정을 로드하고 `pipeline.process_job()`을 호출하는 구조였음.
- **정정**: **`pipeline.py`를 메인 오케스트레이션(진입점)으로, `zighangApi.py`를 직행 전용 어댑터로** 뒤집음.
  - `pipeline.py`: 설정(filters.json) 로드 → `zighangApi.fetch_jobs()` 호출 → 공고별 `zighangApi.fetch_zighang_content()` 호출 → (옵션) AI 분석 → 저장. `python src/pipeline.py`가 실행 커맨드.
  - `zighangApi.py`: `fetch_jobs(config, since, limit)`과 `fetch_zighang_content(url)` 두 공개 함수 + ProseMirror 변환 내부 함수만 남김. 설정 로드·진입점 제거.
- **사유**: 향후 원티드 재도입이나 다른 플랫폼(잡코리아 등) 추가 시, `pipeline.py`의 오케스트레이션 로직을 건드리지 않고 같은 형태(`fetch_jobs`/`fetch_content` 두 함수)의 어댑터 모듈만 병렬로 추가하면 되게 하기 위함.
- **부수 결정**: 두 모듈을 저장소 루트에서 **`src/` 폴더**로 이동(`git mv`). 어댑터 진입점이 아니게 되면서 자연스럽게 소스 코드 폴더 분리 관례를 적용. `filters.json`(§4 참고)·`resume.json`·`jobs/`는 `src/`가 아닌 저장소 루트에 위치하므로, `pipeline.py`에 `ROOT_DIR`(파일 위치 기준 상위 폴더) 상수를 추가해 경로를 고정.
- 워크플로(`job-fresh-pipeline.yml`)의 실행 스텝도 `python src/pipeline.py`로 변경.

### 2. 수집 구간 재설계 — "당일 00시" → "전날 크론 시각부터"

- **기존 결정(UL-0006 §1)**: 크론을 매일 실행하되, **당일 등록 공고만**(KST 00:00 이후) 조회.
- **실측 확인**: 다른 세션에서 직행 API의 실제 필터 형식을 확인함 — `sortCondition=LATEST`(등록순 정렬) + `startDate={KST LocalDateTime}`(이 시각 이후 등록분만 반환). 이 구조에서 크론이 KST 07:00에 실행되는데 조회 시작점을 당일 00:00으로 고정하면, **매일 00:00~07:00 사이 등록된 공고를 영구히 놓치는 공백**이 생김(다음날 조회도 그날 00:00부터라 이 구간은 영원히 재조회되지 않음).
- **정정**: 크론 실행 시각(KST 07:00)은 유지하되, 조회 시작점을 **"전날 KST 07:00"**으로 변경. 즉 매 실행은 직전 실행 이후 전체 구간을 조회하는 rolling window 방식. `COLLECT_START_HOUR = 7` 상수로 관리.
- **dedup 미필요 원칙(UL-0006)과의 관계**: 원래 근거는 "당일 조회 조건이 실행 간 중복을 구조적으로 방지"였음. 조회 조건이 "당일"에서 "직전 실행 이후 구간"으로 바뀌어도, 크론이 정확히 매일 실행되고 구간이 끊김없이 이어지는 한 실행 간 중복 방지 효과는 동일하게 성립하므로 이 원칙 자체는 재확인되며 뒤집히지 않음. 다만 수동 재실행 등으로 구간이 겹칠 경우에 대비해 `pipeline.save_job()`에 파일 id 일치 시 덮어쓰기 처리를 이미 구현해둠(같은 공고 재저장 시 중복 파일 생성 방지).
- **추가 결정(향후 과제)**: 크론의 주기·실행 시각 자체를 코드에 상수로 고정하는 대신, **온보딩 스크립트(`scripts/setup.py`)에서 사용자가 커스터마이징할 수 있게 할 예정**(미구현 — 향후 setup.py 설계 시 반영 대상으로 표시만 해둠).

### 3. frontmatter에 `keywords` 필드 추가

- **발견**: 직행 API 응답을 실측하는 과정에서, 목록 API가 공고마다 자체적으로 부여한 **`keywords` 필드**(문자열 배열, 예: `["네트워크", "장비설정", "장애분석"]`)를 이미 제공하고 있음을 확인함. 이는 우리가 필터링에 쓰는 설정 파일(구 `keywords.json`, §4에서 `filters.json`으로 개명)과는 무관한, API 측이 공고 본문에서 뽑아 부여한 특징 태그.
- **결정**: 이 값을 버리지 않고 frontmatter 필드로 그대로 저장(`keywords: [...]`, `employ_type`과 `collected_at` 사이에 위치). 향후 검색·필터링에 활용할 수 있는 부가 정보이므로 수집 시점에 함께 보존.

### 4. 설정 파일 리네임 — `keywords.json` → `filters.json`, `profile.json` → `resume.json`

- **배경**: §3에서 직행 API 자체의 `keywords` 필드를 frontmatter에 저장하기로 하면서, 우리 설정 파일명이 `keywords.json`이면 두 "keywords"가 이름만으로 혼동될 소지가 생김. 이참에 각 파일의 실제 용법에 맞는 이름으로 정리하기로 함.
- **`keywords.json` → `filters.json`**: 내용물이 직무·지역·경력·채용유형 등 직행 API 수집 필터 조건 집합이므로, 용법을 그대로 드러내는 이름으로 변경.
- **`profile.json` → `resume.json`**: 개인화 분석(personal_comment)에 쓰이는 보유 기술·프로젝트 정보로, "이력서" 성격에 더 부합하는 이름으로 변경.
- 코드 전반(`pipeline.py`의 `FILTERS_SCHEMA_VERSION`·`_RESUME_PATH`·로그 태그·에러 메시지, `.gitignore` 주석) 갱신 완료. `git mv`로 파일 히스토리 유지.

## 파급 확인

- UL-0006 "저장 규칙" 절의 frontmatter 필드 목록에 `keywords`가 누락돼 있었음 — 이번 결정으로 필드 목록에 추가되어야 함.
- UL-0006 "파일 구성" 절의 "`zighangApi.py` — 직행 API 당일 공고 수집, 오케스트레이션" 서술은 §1로 인해 오케스트레이션 주체가 `pipeline.py`로 바뀌었으므로 정정 필요. 두 파일의 경로도 `src/` 하위로 변경됨.
- UL-0006의 "당일 등록 공고만 매일 수집" 서술(기능·아키텍쳐 양쪽)은 §2로 인해 "전날 크론 시각부터 실행 시점까지" 방식으로 정정 필요. dedup 미필요 원칙의 근거 서술도 "당일 조회"에서 "구간 조회(rolling window)"로 갱신 필요하나, 원칙 자체(중복 방지 논리)는 유지됨.
- 문서 전반에서 `keywords.json`·`profile.json`을 언급한 부분(설정 파일 기반 절, 온보딩 스크립트 절, 파일 구성 절)이 전부 `filters.json`·`resume.json`으로 갱신되어야 함.
