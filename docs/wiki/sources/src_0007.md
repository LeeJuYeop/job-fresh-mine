## 배경

UL-0006 확정 후 `zighangApi.py`·`pipeline.py`를 실제 구현하고 GitHub Actions에서 검증하는 과정,

## 최종 결정 사항

### 1. 코드 구조 리팩토링 — 오케스트레이션 반전 + `src/` 폴더 + 설정파일 리네임

- `pipeline.py`를 메인 오케스트레이션(진입점)으로, `zighangApi.py`를 직행 전용 어댑터로 뒤집음
  (설정 로드→`fetch_jobs`→`fetch_zighang_content`→AI→저장). 향후 플랫폼(원티드 재도입,
  잡코리아 등) 추가 시 같은 형태의 어댑터만 병렬로 추가하면 되도록 하기 위함.
- 두 모듈을 `src/` 폴더로 이동. `filters.json`·`resume`·`jobs/`는 루트에 남으므로
  `pipeline.py`에 `ROOT_DIR` 상수를 추가해 경로 고정. 워크플로 실행 스텝도 `python
  src/pipeline.py`로 변경.
- 설정 파일 리네임: `keywords.json`→`filters.json`(수집 필터 조건이라는 용법을 그대로 드러냄,
  §3의 API `keywords` 필드와 이름 혼동 방지 목적도 겸함), `profile.json`→`resume.md`(개인화
  분석용 정보이자 §4에서 자유서식 마크다운으로 최종 확정). `git mv`로 히스토리 유지.
- **구현 상태**: `src/` 이동과 `filters.json` 리네임은 코드 반영 완료. `resume`은 `resume.json`
  리네임까지만 구현됐고, `resume.md`(자유서식) 전환은 §4 결정에 따른 후속 구현 과제로 남음.

### 2. 수집 구간 재설계 — "당일 00시" → "전날 크론 시각부터"

- **문제**: UL-0006은 "당일 등록 공고만"(KST 00:00 이후) 조회로 결정했으나, 직행 API 실측 결과
  `sortCondition=LATEST` + `startDate={LocalDateTime}` 방식임을 확인. 크론이 KST 07:00에
  실행되는데 조회 시작점을 당일 00:00으로 고정하면 매일 00:00~07:00 등록분을 영구히 놓치는
  공백이 생김.
- **정정**: 크론 실행 시각(07:00)은 유지하되 조회 시작점을 "전날 KST 07:00"으로 변경(rolling
  window). `COLLECT_START_HOUR = 7` 상수로 관리.
- **dedup 미필요 원칙과의 관계**: 근거가 "당일 조회"에서 "직전 실행 이후 구간 조회"로 바뀌어도,
  크론이 끊김없이 매일 실행되는 한 실행 간 중복 방지 효과는 동일하게 성립 — 원칙 자체는 유지됨.
  수동 재실행 등 구간이 겹치는 경우를 대비해 `save_job()`에 id 일치 시 덮어쓰기 처리를 구현해둠.
- 크론 주기·시각 자체의 커스터마이징은 §4의 온보딩 경로(정적 생성기 사이트/설정 도구)에서
  추후 지원할 예정(미구현).

### 3. 저장 포맷 확장 — frontmatter `keywords`·`deadline`·`end_date` 필드, 파일명 마감 표기

- **keywords**: 직행 목록 API가 공고마다 자체 부여한 특징 태그(`["네트워크", "장비설정", ...]`,
  우리 필터 설정과 무관)를 확인. 향후 검색·필터링 활용을 위해 frontmatter에 그대로 보존
  (`employ_type`과 `collected_at` 사이).
- **마감 정보**: `companyTypes`(기업규모 7종)·`deadlineTypes`(마감유형: 마감일·상시채용·
  채용시마감)가 반복 파라미터 필터로 지원됨을 실측 확인(레거시의 "deadlineType 미지원" 주석은
  단수형 오판이었음). `filters.json`에 두 필드 추가, `fetch_jobs()` 필터 목록에 반영.
- 마감 정보는 저장 파일에도 반영: 파일명을 `[회사명][경력][채용유형][지역][마감] {title}.md`로
  확장하고 frontmatter에 `deadline`·`end_date` 추가. `deadline` 값은 마감일 타입이면
  `YY-MM-DD마감`(절대 날짜 — 수집 시점 기준으로 어긋나는 상대 표기 대신 채택), 그 외는
  유형명 그대로. `end_date`는 마감일 타입만 값 존재.
- **한계**: 기업규모는 응답의 company 객체에 없어 필터 전용(공고별 저장 불가).

### 4. 온보딩 개선 — filters·resume 정적 생성기 사이트(GitHub Pages) 도입, resume 형식 전환

- `filters.json` 값은 직행 API의 고정 열거값이라 자유 텍스트 편집 시 잘못된 값 위험이 큼.
  AWS 콘솔 정책 생성기(체크박스 선택→JSON 자동 생성)를 참고해, 저장소가 소유한 정적 사이트를
  만들기로 함. 사용자는 체크박스·드롭다운으로 직무·지역·경력·기업규모·마감유형 등을 선택해
  결과를 생성하고, 복사해 GitHub 웹 에디터로 `filters.json`에 붙여넣어 커밋한다. 선택지는
  직행 실제 데이터를 참고해 하드코딩.
- `resume`도 같은 사이트에서 지원: 기술스택 등 선택 가능한 항목은 하드코딩 picklist로, 프로젝트
  이력 등은 자유 서술 입력란으로 제공해 조합 생성. `resume`은 실제로는 파이썬 파서가 아니라
  Lambda AI 프록시의 프롬프트 컨텍스트이므로 구조화 JSON보다 자연어가 유리하다고 판단—
  `resume.md`(자유서식 마크다운)로 최종 형식을 확정하고 깃에 그대로 커밋(민감 개인정보 아님).
  생성기 사이트가 최소한의 구조·글자수 유도 역할을 겸함.
- `scripts/setup.py`는 폐기하지 않되 필수 온보딩 경로에서 제외하고 파워유저용 도구로 격하.
  `pipeline.py`의 `filters.json` 스키마 검증은 손편집 등 우회 경로 대비 2차 방어선으로 유지.
