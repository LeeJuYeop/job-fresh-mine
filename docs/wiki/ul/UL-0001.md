## 현재결정사항

Gongtion은 한국 채용공고를 자동 수집·AI 분석하여 Notion DB에 저장하는 GitHub Actions 기반 파이프라인이다. README.md(2026-06-17 커밋 기준)에 문서화된 시스템의 현재 상태는 다음과 같다.

**주요 기능**
- 자동 크롤링: 직행(공식 API), 원티드(비공식 JSON API)에서 공고 수집. Notion 링크 필드를 기준으로 중복 자동 방지.
- AI 구조화: Gemini 2.5 Flash가 공고 본문에서 직무·기술스택·경력·채용유형을 추출하고 한줄 요약 코멘트 생성.
- 개인화 분석(선택): `profile.json`에 보유 기술·프로젝트를 작성하면 공고와의 매칭도를 Notion 페이지에 추가.
- Notion 저장: 구조화 필드는 DB Properties로, AI 요약과 공고 본문은 페이지 블록으로 저장.
- 설정 파일 기반: `keywords.json` 수정만으로 검색 키워드·직무·지역·경력 필터 변경 (코드 수정 불필요).

**동작 원리**
- GitHub Actions 크론(화·수·토 07:00 KST, 커스터마이징 가능) → `crawler.py` 실행.
- 직행: 공고 목록 수집 → ProseMirror 본문 추출 → Gemini(기술스택만 추출) → Notion 저장.
- 원티드: 공고 목록 수집 → 상세 API 본문 추출 → Gemini(직무·경력·채용유형·기술스택 추출) → Notion 저장.
- 회사명·공고명·지역은 API 응답에서 직접 주입하여 Gemini 할루시네이션 방지.
- API 본문 추출 실패 시 Jina Reader로 자동 폴백.
- 실행당 처리 한도: 직행 8건, 원티드 2건 (API 과금 대응).
- `keywords.json`의 `zighang`/`zighang_weekend` 필터를 상황별로 다르게 적용 가능하며, 스케줄·필터 전환 조건은 GitHub Actions 설정으로 커스터마이징 가능.

**Notion DB 스키마**
| Property | 타입 | 비고 |
|---|---|---|
| 회사명 | Title | |
| 공고명 | Rich text | |
| 직무 | Multi-select | 서버_백엔드, DevOps_SRE 등 |
| 기술스택 | Multi-select | Gemini 추출 |
| 경력 | Select | 신입 / 경력 / 무관 |
| 채용유형 | Select | 정규직 / 인턴 |
| 지역 | Multi-select | 시/도 단위 |
| 링크 | URL | 중복 방지 기준 필드 — 이름 정확히 일치 필수 |

페이지 본문 구조: AI 요약 Callout(🤖) → 개인화 분석 Callout(👤, 선택) → 공고 본문 마크다운.

**파일 구성**
- `crawler.py` — 사이트별 URL 수집, 중복 확인, 오케스트레이션
- `pipeline.py` — 본문 추출 → Gemini 분석 → Notion 저장 공통 파이프라인
- `keywords.json` — 검색 키워드·직무·지역·경력 필터 설정
- `profile.json` — 개인화 분석용 프로필 (선택)
- `requirements.txt` — Python 의존성
- `.github/workflows/crawler.yml` — 크론 스케줄 및 Secrets 주입

**Fork 사용 절차 (요약)**
저장소 Fork → Notion Integration 생성 및 DB 구성(`NOTION_API_KEY`, `NOTION_DATABASE_ID`) → Gemini API Key 발급(`GEMINI_API_KEY`) → GitHub Secrets 등록 → `keywords.json` 설정(`depthTwos` 허용값: 서버_백엔드/DevOps_SRE/시스템_네트워크/시스템소프트웨어/웹풀스택) → `profile.json` 선택 설정 → GitHub Actions 활성화(Fork 저장소는 기본 비활성) 및 수동 실행 테스트.
