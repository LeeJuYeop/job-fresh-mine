## 현재결정사항

친구들에게 Gongtion을 사용해보게 하려 했으나, 초기 세팅 절차(Fork → Notion DB 스키마 생성 → Notion Integration·API 키 발급 → DB 연결 → keywords.json 편집 → Gemini API 키 발급 → GitHub Secrets 등록 → Actions 수동 활성화)가 지나치게 많아 이탈률이 높을 것으로 판단되어, "자동화" 컨셉은 유지한 채 배포 아키텍처를 다음과 같이 재설계하기로 결정함.

1. **실행 모델**: 분산형 fork 구조를 유지한다 (중앙 실행 모델로 전환하지 않음).
2. **저장소**: Notion DB 저장을 폐기하고, 각 fork 저장소 내부 Git 파일 저장으로 전환한다. 이에 따라 Notion 관련 셋업 절차(DB 생성·Integration 생성·API 키 발급·DB 연결) 전체가 제거된다.
3. **AI 기능**: 파이프라인의 필수 코어에서 분리하여 옵션 기능으로 재배치한다. 사용자별 Gemini API 키 발급·등록 방식을 폐기하고, 운영자가 소유한 AWS Lambda 공용 프록시가 Gemini 호출을 대행한다(운영자의 Gemini 키는 Lambda 환경변수에만 존재하며 사용자 fork에는 AI 관련 시크릿이 필요 없음).
4. **레이트리밋 대응**: 다수 사용자의 크론이 비슷한 시간대에 몰려 Gemini 무료 티어 레이트리밋(분당 15회)에 걸리는 문제를, 다음 조합으로 해소한다 — 크론 지터(온보딩 스크립트가 ±0~25분 랜덤 오프셋 자동 배정) + Lambda 호출 직전 0~5분 랜덤 sleep + Lambda Reserved Concurrency 1~2로 전역 직렬화 + Lambda 내부 토큰버킷(분당 12회 자체 상한)과 429 시 지수 백오프 재시도(1s→2s→4s, 2~3회) + 여러 공고를 배치로 묶어 호출 수 절감 + 재시도까지 실패 시 요약 없이 진행(최후 폴백). 이 규모(5명 내외, 약 1개월 운영)에서 SQS 등 큐 인프라는 과설계로 판단하여 도입하지 않는다.
5. **인증**: 사용자 fork에 AI 관련 시크릿을 요구하지 않는 것을 목표로, GitHub OIDC(Actions `id-token: write` + API Gateway JWT Authorizer로 issuer/repository claim 검증) 방식을 우선 검토한다. 구현 난이도가 높을 경우 저위험 정적 프록시 키(Gemini 키와 무관) 하나만 Secrets에 등록하는 절충안을 대안으로 둔다.
6. **개인화 분석 기능**: `profile.json` 기반 개인화 분석(보유 기술·프로젝트와 공고 매칭도 비교) 기능 자체는 축소·폐기 없이 그대로 유지한다. 매칭 결과를 기존에는 Notion 페이지 블록에 추가했다면, 저장 매체 전환에 맞춰 Git 파일(마크다운)에 함께 기록하는 것으로 저장 매체만 대체한다.

## Basis

UL-0001은 README.md에 기록된 시스템의 현재 상태(Notion 저장, 사용자별 API 키 개별 발급)를 있는 그대로 기록한 베이스라인이었을 뿐, "왜 Notion을 쓰는지" 또는 "왜 개인별 키 발급 방식인지"에 대한 별도의 설계 근거를 주장하지 않았다. 따라서 이번 UL이 그 방식들을 대체하는 것은 UL-0001의 Basis와 상충하지 않는다 — UL-0001은 단순 현황 기록이었고, 이번 UL은 "지인 대상 배포 시 온보딩 이탈률이 너무 높다"는 새로운 문제의식에서 출발한 결정이다.

Notion 저장을 폐기하는 이유는 (a) DB 한 줄 = 페이지 1개 구조라 방치 시 조회 성능이 저하되고 전체 삭제 기능이 없다는 사용성 문제, (b) Notion 관련 셋업 자체가 진입장벽의 상당 부분을 차지한다는 온보딩 문제 두 가지다. AI 기능을 공용 Lambda 프록시로 옮기는 이유는, 사용자별 Gemini 키 발급이 온보딩 절차의 남은 병목이자, 이를 없애기 위해 운영자 키를 각 fork에 공유 배포하면 여러 사람이 통제하지 못하는 저장소에 같은 키가 흩어지는 유출 위험이 생기기 때문이다 — 이 두 문제를 모두 해소하는 절충점이 "얇고 stateless한 공용 프록시"다. 이 프록시는 크롤링·중복방지·Git 저장이라는 핵심 파이프라인과 분리되어 있어(AI 실패 시에도 핵심 기능은 영향받지 않음), 실행 모델을 분산형(fork) → 중앙형으로 전환하는 것은 아니다.

## Living Spec Update Log

living-spec.md의 3개 섹션을 이번 결정에 맞춰 갱신함:
- **기능**: "Notion 저장" 항목을 "Git 저장"으로 대체. "AI 구조화"를 옵션 기능으로 표시하고 AWS Lambda 프록시 경유로 서술 변경. "개인화 분석(선택)"은 결과 저장 매체를 Notion 페이지 → Git 파일로 갱신(기능 자체는 유지). Notion DB 스키마 표는 저장소가 Git으로 바뀌었으므로 제거.
- **아키텍쳐**: 실행 모델이 분산형 fork 유지임을 명시하는 문장 추가. 직행/원티드 처리 흐름의 종착점을 "Notion 저장" → "Git 저장"으로 변경하고 AI 호출을 "(옵션) AI 프록시 경유"로 수정. 크론 스케줄에 자동 지터 배정 사실 추가. 신규 서브섹션 "AI 프록시 및 레이트리밋 대응"을 추가하여 Lambda 프록시 구조, 레이트리밋 대응 조합, 인증 방식(GitHub OIDC 우선)을 기록. 파일 구성 설명에서 pipeline.py·crawler.yml 설명을 갱신.
- **기술스택**: "Notion API (저장소)"를 "Git (파일 기반 저장소)"로 대체, "AWS Lambda (AI 공용 프록시)" 신규 추가. 필수 환경변수를 "사용자 fork 측(목표: 없음)"과 "운영자(Lambda) 측(GEMINI_API_KEY)"으로 분리 기술.

index.md에는 UL-0002를 UL 목록에 등재함(신규 category 없음, 기존 3개 category 재사용).
