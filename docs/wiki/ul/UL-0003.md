## 현재결정사항

UL-0002에서 확정된 "온보딩 장벽 해소" 방향(Git 저장, AI Lambda 프록시)을 구체적으로 실행하기 위한 **로컬 온보딩 스크립트 `scripts/setup.py`**의 설계를 확정한다. 사용자는 Fork → Clone 후 이 스크립트를 1회 실행하는 것만으로 keywords.json·profile.json 설정을 마친다.

**keywords.json 자동화**
1. 스크립트 실행 시점에 직행 API로 필터 없는 조회를 보내 현재 유효한 `depthTwos`/`regions`/`employeeTypes` 태그 전체를 실시간 수집한다(크론이 아니라 setup 시점에만 — 페이지네이션 없는 단일 요청 수준이라 직행 서버에 부담 없음). 태그 매핑을 코드에 영구 하드코딩하지 않아, 직행이 태그 체계를 바꿔도 다음 setup 실행 시 자동 반영된다.
2. 수집한 태그를 체크박스 picklist로 보여주고 선택받아, 원본 태그 문자열 그대로 `keywords.json`에 기입한다. 사용자는 원본 태그 문자열을 직접 타이핑하지 않는다.
3. 생성된 `keywords.json`을 `git add/commit/push`로 자동 반영한다(plain git, 사용자의 기존 git 자격증명 재사용). push 전 `origin`이 본인 fork를 가리키는지 확인하는 안전장치를 둔다.

**profile.json 자동화(선택 기능)**
4. "개인화 매칭 기능을 쓰시겠습니까? (y/N)" opt-in 질문으로 시작하며, 원치 않으면 이 섹션 전체를 건너뛴다(파일 없으면 개인화 기능 자동 비활성화라는 기존 동작과 일치).
5. `tech_stack`/`learning_interests`는 쉼표 구분 자유 텍스트 한 줄 입력을 받아 `split(",")` 후 trim만 하고 저장(정답이 없는 개인 데이터라 화이트리스트 검증 없음).
6. `projects`는 이름/설명/사용기술 질문을 반복 루프로 받아 여러 개를 쌓는다(빈 입력 시 자동 종료).
7. 사용자가 JSON 문법을 직접 다루지 않도록 스크립트가 `json.dump`로 생성하고, 생성 직후 `json.load`로 재검증한다.
8. 재편집 소유권은 사용자에게 넘긴다 — 스크립트는 최초 1회 스캐폴딩만 돕고, 이후에는 사용자가 직접 텍스트 에디터로 수정. 이를 보완하기 위해 주석이 달린 `profile.json.example` 샘플 파일을 별도로 둔다.
9. fork 저장소가 public이면 프로젝트 설명 등 개인 데이터가 git 히스토리에 노출되므로, README에 경고 문구를 추가한다.

**`gh` CLI 의존성 완전 배제**
10. `gh api`(Actions 자동 활성화), `gh secret set`(시크릿 자동 등록), `gh workflow run`(셀프테스트 자동화)을 스크립트에서 전부 제외한다 — 스크립트가 사용자 GitHub 계정 쓰기 권한을 갖는 순간 결함 하나가 여러 계정에 반복 영향을 미치는 위험이 생기기 때문.
11. Actions 활성화는 GitHub 웹 UI에서 수동 2클릭으로 처리(기존 README 절차와 동일 수준).
12. AI 관련 시크릿은 GitHub OIDC 구현으로 확정하여 애초에 등록 대상이 없다(UL-0002에서 "우선 검토"로 열어두었던 것을 이번에 확정).
13. 셀프테스트는 자동화하지 않고 옵션(수동 "Run workflow" 클릭)으로만 남긴다. 다만 "Actions 비활성화" 상태는 에러 로그 없는 침묵 실패이므로, README 트러블슈팅 섹션과 `setup.py` 종료 시 "다음 할 일 체크리스트" 출력으로 보완한다.

## Basis

UL-0002는 "AI 관련 시크릿 인증 방식"을 GitHub OIDC 우선 검토, 정적 프록시 키를 대안으로 열어둔 조건부 결정이었다. 이번 UL은 그 조건부 상태를 OIDC로 확정 짓는다 — 이는 UL-0002의 Basis(사용자 fork에 AI 관련 시크릿을 요구하지 않는 것이 목표)와 상충하지 않고 오히려 그 목표를 완전히 달성하는 방향이다.

`gh` CLI를 온보딩 스크립트에서 배제하기로 한 것은 새로운 판단이다: 스크립트가 사용자의 GitHub 계정에 대한 쓰기 권한(Secrets 등록, Actions 상태 변경)을 갖는 자동화는, 스크립트 결함이 여러 사용자 계정에 걸쳐 반복적으로 영향을 미치는 위험한 형태가 된다. Actions 활성화·셀프테스트처럼 실패해도 사용자가 스스로 인지하고 웹 UI에서 수동으로 되돌릴 수 있는 저위험 조작은 자동화 대신 수동 절차로 남기는 것이, "약간의 수동 마찰"과 "자동화가 가진 계정 권한 위험"을 저울질했을 때 이 규모(지인 수명, 1개월 운영)에 더 적합하다.

profile.json 입력을 온보딩 스크립트에 통합하는 것은 keywords.json과 근본적으로 다른 성격의 문제(정답이 있는 매칭 문제 vs 정답이 없는 자기 서술 문제)에 대한 결정이다. 자기 서술 비용 자체는 없앨 수 없지만, 이미 존재하는 setup.py에 opt-in 질문 몇 개를 추가하는 것은 한계 비용이 낮아, "이 정도는 문서화(example 파일)만으로 충분하다"는 관점과 "스크립트가 JSON 문법 오류를 원천 차단해주는 것이 낫다"는 관점을 모두 수용하는 절충으로 결정했다 — 두 관점이 상충하는 것이 아니라 각각 다른 비용(초기 입력 마찰 vs 새 인프라 구축 비용)을 지적한 것이므로, 저비용인 대화형 프롬프트 추가와 저비용인 example 템플릿 제공을 함께 채택했다.
