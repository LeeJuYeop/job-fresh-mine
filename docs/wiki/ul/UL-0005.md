## 현재결정사항

새 프로젝트 착수 전 레거시(Gongtion 시절) 잔재를 4가지 축으로 정리한다.

1. **프로젝트명 변경**: `Gongtion` → `job-fresh`.
2. **용어 변경(크롤링 → API)**: 실질이 REST API 호출에 가까우므로 "크롤링" 용어를 "API"로 전환. 파일명도 함께 변경: `crawler.py` → `zighangApi.py`(완료, 원본은 `legacy_crawler.py`로 보존), `.github/workflows/crawler.yml` → `job-fresh-pipeline.yml`(결정 사항, 실제 반영은 예정 — 현재는 `legacy_crawler.yml`로만 보존된 상태).
3. **플랫폼 축소(원티드 배제, 직행 단일화)**: 원티드(비공식 JSON API)를 배제하고 직행(공식 API)에만 집중. 근거는 직행 집중을 통한 개발 범위 축소(원티드 API 자체의 불안정성이 주된 사유는 아님). 이에 따라 원티드를 사용하지 않으므로 본문 추출 폴백 수단이었던 Jina Reader도 함께 폐기한다.
4. **living-spec.md 세부 서술 삭제**: 아키텍쳐 섹션의 "실행당 처리 한도: 직행 8건, 원티드 2건 (API 과금 대응)" 문장 전체를 삭제하고(직행 수치 포함), "`keywords.json`의 `zighang`/`zighang_weekend` 필터를 상황별로 다르게 적용..." 문장도 삭제한다. 후자의 삭제는 단순 문서 정리가 아니라 `zighang_weekend`(요일별 지역 필터) **기능 자체의 폐기**를 의미하며, `keywords.json`의 `zighang_weekend` 섹션과 `legacy_crawler.py`의 요일별 분기 로직(`resolve_zighang_cfg`)도 함께 폐기 대상이다. 이후로는 `zighang` 단일 설정만 사용한다. 폐기 사유는 요일별로 지역 필터를 다르게 가져갈 필요 자체가 없다고 판단했기 때문이다.
5. **처리 한도(비용 통제) 재설계는 Pending**: "API 과금 대응"이라는 원래 근거 자체를 부정하는 것은 아니며, 비용 통제 필요성은 유효하다. 다만 구체적 수치(예: 직행 N건)는 지금 living-spec에 다시 못박지 않고 코드 재작성 시점에 재설계하기로 하여 **미결(Pending)**로 남긴다.
6. **직행 API 본문 추출 실패 시 처리**: Jina Reader 폐기에 따라 직행 API 자체의 본문 추출이 실패한 경우도 별도 폴백 없이 실패로 처리한다 — 해당 공고는 건너뛴다(`legacy_crawler.py`의 `fetch_zighang_content`가 전제했던 Jina 폴백 경로는 사용하지 않는다).

## Basis

- **프로젝트명·용어 변경(2)**: UL-0001은 README 기준 현황을 있는 그대로 기록한 베이스라인일 뿐, "왜 이름이 Gongtion인지" 또는 "왜 크롤링이라는 용어를 쓰는지"에 대한 별도 설계 근거를 주장하지 않았다. 따라서 이번 이름·용어 변경은 과거 Basis와 상충하지 않는다 — 새 프로젝트 시작 시점에 실체(REST API 호출)에 맞게 표현을 바로잡는 단순 정정이다.
- **원티드 배제(3)**: UL-0001 베이스라인과 UL-0002는 직행·원티드 두 플랫폼 사용을 현황/전제로 다뤘을 뿐, "왜 두 플랫폼을 모두 써야 하는지"에 대한 적극적 Basis를 주장한 적이 없다(UL-0002의 Basis는 저장 매체 전환과 AI 프록시 도입에 집중되어 있었고 플랫폼 구성 자체는 건드리지 않았다). 따라서 원티드를 배제하는 이번 결정은 과거 Basis를 뒤집는 것이 아니라, 지금까지 명시적으로 다뤄진 적 없던 "플랫폼 범위"에 대한 신규 결정이다.
- **Jina Reader 폐기 및 직행 자체 폴백 부재(6)**: 원티드 배제의 직접적 파생 결과다. `legacy_crawler.py`의 `fetch_zighang_content`도 API 본문 추출 실패 시 Jina로 폴백하는 경로를 갖고 있었으나(UL-0005-qa.md Q3 답변), 이번 결정으로 직행 전용 구성에서도 Jina 폴백을 두지 않으며 추출 실패 시 해당 공고를 건너뛰는 것으로 명시적으로 확정한다. 안정성보다 구성 단순화를 우선한 결정이다.
- **zighang_weekend 폐기(4)**: UL-0001 베이스라인이 기존 기능으로 기록했던 것을 이번에 명시적으로 폐기한다. 폐기 근거는 요일별로 지역 필터를 다르게 가져갈 필요 자체가 없다는 판단이다(UL-0005-qa.md Q2 답변). UL-0001은 이 기능에 대해 적극적 Basis를 주장한 바 없어(선행 UL 없는 최초 현황 기록), 이번 폐기는 과거 Basis를 뒤집는 것이 아니라 처음으로 그 필요성을 재평가한 결정이다.
- **처리 한도 문장 삭제, 비용 통제 근거는 Pending(5)**: 삭제되는 문장의 원래 Basis는 "API 과금 대응"(비용 통제, UL-0001 베이스라인)이었다. 이번 결정은 비용 통제의 필요성 자체를 부정하지 않는다 — 다만 구체적 수치(직행 8건 등)를 지금 다시 정하지 않고 새 코드 재작성 시점으로 미루기로 했다(UL-0005-qa.md Q1 답변). 따라서 과거 Basis는 "해소"가 아니라 "재설계 보류" 상태이며, index.md Pending 목록에 등재해 추적한다.
