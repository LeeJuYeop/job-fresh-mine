## 현재결정사항

UL-0005 확정 후 코드 이행 착수 전, Code Reviewer 에이전트의 레거시 이행 리뷰로 식별된 설계 공백 8개와 논의 중 추가된 AI 범위 축소 1개를 다음과 같이 확정한다.

1. **일일 수집 전환**: 화·수·토 07:00 KST 크론을 폐기하고 **매일 실행**한다. 매 실행은 직행 API로 **당일 등록 공고만** 조회한다(API 지원 확인됨). keywords.json 기반 직무·경력 등 필터 원칙은 유지. 탐색 개수는 **하루 20건(잠정)** — 확정 수치는 추후 결정(UL-0005의 처리 한도 Pending은 이 잠정치로 운영하며 계속 열어둠).
2. **dedup 폐기**: "당일 등록분만 조회" 조건으로 실행 간 중복이 구조적으로 발생하지 않으므로 별도 중복 확인 로직을 두지 않는다. 동일 공고의 여러 날 재게시는 거르지 못함을 인지한 채 수용한다.
3. **Git 저장 포맷**: 공고당 1파일, YAML frontmatter + 마크다운 본문. 저장 경로는 **`jobs/{YYYY-MM-DD}/` 일별 폴더**(확인한 공고와 안 한 공고를 날짜로 구분). 파일명은 **`[회사명][경력][채용유형][지역] {title}.md`**(정보 최대 표시; Windows 금지문자 제거·길이 제한, 복수 지역은 첫 지역만 파일명에, 전체는 frontmatter에). frontmatter에 `schema_version` 명시(UL-0004 이행).
4. **14일 삭제 원칙**: 14일 이상 지난 일별 폴더는 매 실행 시 워크플로가 폴더째 삭제한다.
5. **고유 ID**: frontmatter `id` 필드에 `zighang-{직행 recruitment UUID}` 형식으로 기록(UL-0004 이행 방식 확정). dedup 폐기로 ID 용도는 재처리·추적·Claude 루틴 레시피의 전제로 축소.
6. **AI 가공 범위 축소 — 기술스택 추출 폐지**: AI는 **한줄요약(ai_comment)만** 수행한다. 기술스택 추출은 Notion DB multi-select 표시용이었으므로 Notion 폐기(UL-0002) 후에는 불필요 — 본문에 이미 있는 내용의 재정리일 뿐. 개인화 매칭(personal_comment)은 유지.
7. **Lambda AI 프록시 계약**: 프롬프트는 Lambda가 소유하고 fork는 공고 데이터(+선택 시 profile)만 구조화 JSON으로 전송 — 임의 프롬프트 남용 차단, 프롬프트 버전 운영자 일괄 관리. **profile.json 전송 허용**하되 Lambda 무로깅 원칙을 README에 명시.
8. **커밋/푸시 방식**: 기본 `GITHUB_TOKEN` + `permissions: contents: write`로 기본 브랜치 직접 커밋·푸시(추가 시크릿 없음). `concurrency` 그룹 + push 전 `git pull --rebase`로 충돌 방지. 일별 폴더 생성·14일 폴더 삭제도 이 커밋에 포함.
9. **OIDC 검증 규칙**: 운영자 관리 **허용 리스트**(사용자 계정 목록, Lambda 환경변수 등) — 이름만 맞춘 무관 저장소의 무임승차 차단. 규모 확장 시 패턴 매칭으로 완화 검토.
10. **AI 실패 표식**: frontmatter `ai_status: done | skipped`. 재처리 파이프라인은 구현하지 않음(과설계). 14일 삭제 원칙상 재처리 기회도 자연히 14일로 제한.
11. **keywords.json 스키마**: 평탄화 + `schema_version` — `zighang` 래핑 제거, 최상위에 `depthTwos`/`regions`/`employeeTypes`/`careerMin`/`careerMax`/`educations` 직접 배치(직행 단일화 확정으로 래핑 불필요).

## Basis

- **일일 수집 전환(1)**: UL-0001 베이스라인의 화·수·토 크론은 적극적 근거 없이 기록된 현황이었다. 매일 실행 + 당일분만 조회는 "확인한/안 한 공고의 날짜별 구분"이라는 새 사용성 요구와 dedup 제거(2)를 동시에 충족한다. UL-0002의 크론 지터·레이트리밋 대응 Basis는 매일 실행에서도 그대로 유효하나, 실행 빈도가 주 3회→7회로 늘어나는 점은 인지하고 수용한다(회당 처리량이 하루 20건으로 제한되므로 Gemini 무료 티어 내 운용 가능 판단).
- **dedup 폐기(2)**: UL-0002는 "Git 저장소 내 기존 저장 항목 기준 중복 자동 방지"를 기능으로 확정했었다. 이번 결정은 그 기능 자체를 폐기하는 것이 아니라 **구현 수단을 대체**한다 — 당일 등록분만 조회하는 수집 조건이 실행 간 중복 방지 역할을 흡수하므로 별도 로직이 불필요해졌다. 다만 여러 날 재게시 중복까지 막던 원래 서술보다 보장 범위가 좁아지는 것은 사실이며, 이 축소를 명시적으로 수용한다.
- **저장 포맷·14일 삭제(3·4)**: UL-0002(Git 파일 저장 전환)와 UL-0004(스키마 버전·고유 ID)가 요구만 하고 형식을 정하지 않았던 부분의 구체화 — 과거 Basis와 상충 없음. 14일 삭제는 새 원칙으로, "저장 파일이 무한 누적되면 안 된다"는 운영 판단이며 Git 히스토리에는 남으므로 완전 소실은 아니다.
- **`schema_version`과 UL-0004의 "호환성 비보장(best-effort)" 원칙의 관계(3)**: 두 결정은 상충하지 않는다. 버전 필드는 스키마 **변경을 감지할 수단**을 제공하는 것이지 하위 호환을 **보장하는 계약**이 아니다 — 오히려 "버전이 바뀌면 형식이 바뀐 것"을 외부(Claude 루틴 등)가 알 수 있게 해 best-effort 원칙을 실행 가능하게 만드는 메커니즘이다. UL-0004가 README 레시피에 요구한 비보장 명시 의무는 그대로 유효하며, schema_version 도입이 이를 대체·완화하지 않는다.
- **14일 삭제와 UL-0004의 Claude 루틴 레시피의 관계(4)**: 삭제 정책은 레시피가 읽을 수 있는 1차 소스의 유효 시간창을 14일로 제한한다. 다만 루틴 시나리오의 통상 대상은 당일~최근 공고이므로 실질 충돌은 낮고, 14일을 넘긴 파일도 Git 히스토리 조회로 접근 가능하다. 이 제약은 레시피가 어차피 감수해야 하는 best-effort 성격(UL-0004)의 일부로 본다.
- **고유 ID 방식(5)**: UL-0004는 예시로 "공고 URL 해시 등"을 들었으나 방식을 확정하지 않았다. 직행 네이티브 UUID 채택은 그 예시와 다른 선택이지만 UL-0004의 목적(안정적 고유 식별)을 동일하게 달성하며, 직행 단일화(UL-0005) 확정으로 플랫폼 중립성 요구가 사라져 해시의 이점이 소멸했기 때문이다. UL-0004가 ID의 전제로 삼았던 dedup 용도는 (2)로 축소되나, 재처리·추적 용도는 유지된다.
- **기술스택 추출 폐지(6)**: UL-0001 베이스라인에서 기술스택 추출의 근거는 Notion DB multi-select 표시였다. UL-0002가 Notion을 폐기하면서 이 근거는 이미 소멸했으나 기능은 관성적으로 living-spec에 남아 있었다(UL-0005에서도 추출 필드를 기술스택만으로 축소하며 유지). 이번에 그 관성을 끊는다 — 과거 Basis(Notion 표시용)가 소멸했음을 확인하고 폐기하는 것이므로 모순이 아니라 정리다. 이로써 AI 경로의 공식 산출물은 한줄요약(+선택적 개인화 매칭)만 남는다.
- **Lambda 계약(7)**: UL-0002(공용 프록시)·UL-0004(운영 시임)가 정한 방향의 구체화. 프롬프트를 Lambda가 소유하는 것은 "운영자가 유일하게 관찰·개입 가능한 지점"이라는 UL-0004의 Basis를 프롬프트 관리까지 확장 적용한 것. profile 전송 허용은 UL-0002의 "개인화 기능 유지" 결정을 프록시 구조에서 실현하기 위한 필요조건이며, 새로 생기는 프라이버시 우려(개인 데이터의 운영자 인프라 경유)는 무로깅 원칙 명시로 대응한다.
- **커밋/푸시(8)·OIDC(9)**: UL-0002의 "사용자 fork에 시크릿을 요구하지 않는다"와 UL-0003의 "스크립트에 과도한 계정 권한을 주지 않는다" Basis를 그대로 계승한 선택 — GITHUB_TOKEN은 워크플로 기본 제공 권한이라 추가 등록이 없고, 허용 리스트는 운영자 측 관리라 사용자 부담이 없다.
- **AI 실패 표식(10)**: UL-0002의 최후 폴백("요약 없이 진행")으로 저장된 레코드의 식별 수단. backfill 미구현은 UL-0004의 "죽은 추상화를 만들지 않는다"는 보수적 기조와 일치.
- **keywords.json 평탄화(11)**: UL-0003의 setup.py 설계는 `zighang`/`zighang_weekend` 두 블록 전제였으나 UL-0005가 weekend를 폐기했고, 직행 단일화로 남은 단일 블록의 래핑도 의미를 잃었다. setup.py는 이 평탄화 스키마를 생성 대상으로 삼는다(설계 자체는 유지, 출력 스키마만 갱신).
- **처리 한도 잠정치(1)**: UL-0005가 Pending으로 남긴 "처리 한도 구체 수치 재설계"에 대해 잠정 운영값(하루 20건)을 정하되, 확정 수치 결정은 계속 미룬다 — Pending은 해소되지 않고 유지된다.
